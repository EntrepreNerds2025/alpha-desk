# Tech Stack and Hosting

## Stack decisions

### Frontend
- **Framework**: Next.js 15 (App Router)
- **UI kit**: shadcn/ui + Tailwind CSS 4
- **Charts (outside TradingView)**: Recharts
- **State**: Zustand for client state, React Query for server data, Supabase realtime subscriptions
- **Auth**: Supabase Auth (email + optional Google/GitHub OAuth)
- **Generated by**: Lovable, then refined in Claude Code / Codex
- **Host**: Vercel

### Backend / Database
- **Database**: Supabase (Postgres + Realtime + Auth + Storage + Vault for secrets)
- **Edge functions**: Supabase Edge Functions for webhooks and public APIs
- **Migrations**: SQL files in `/supabase/migrations`, applied via Supabase CLI

### Execution Worker (always-on)
- **Runtime**: Node.js 20 (TypeScript)
- **Framework**: Fastify (fast, small, typed)
- **Queue**: BullMQ (Redis-backed) for event processing
- **Host**: Fly.io, `ord` region (Chicago — close to CME data center in Aurora, IL)
- **Redis**: Upstash serverless Redis ($0 tier)
- **Keep-alive**: health check endpoint + Fly.io auto-restart
- **Broker SDK**: `tradovate-js-sdk` (community) or raw REST/WebSocket

### AI layer
- **Providers**: OpenAI + Anthropic simultaneously
- **Routing**: Internal `ai-routing` package picks provider/model by agent + task type
- **Caching**: Anthropic prompt caching for Sonnet system prompts; OpenAI cached input for GPT-5.4 mini repeat context
- **Observability**: Log every call to `ai_messages` table with cost

### Monitoring
- **Logs**: Fly.io logs + Supabase logs (V1) → Axiom or Better Stack (V2)
- **Errors**: Sentry on both frontend and worker
- **Uptime**: Better Stack uptime monitors on webhook endpoint and worker health

## Environments

### Local dev
- `pnpm dev` starts web (Vercel-compatible) + worker
- ngrok for TradingView webhook → local worker
- Supabase local (`supabase start`) OR hosted dev project

### Staging
- Branch: `staging` → Vercel preview + Fly.io staging app + Supabase staging project
- Paper broker only
- Use staging to test strategy changes before promoting

### Production
- Branch: `main` → Vercel production + Fly.io production app + Supabase production project
- Paper broker in Auto Paper mode
- Live broker only enabled per-workspace, per-user confirmation

## Secrets management

**Nothing sensitive in frontend.** Ever.

### Where secrets live
- TradingView webhook secret → Supabase Vault, read by Edge Function webhook handler
- Tradovate API keys → Supabase Vault, read by worker
- OpenAI/Anthropic API keys → Fly.io secrets + Vercel env for Edge Functions
- Supabase service role key → Fly.io secrets (worker only)
- Supabase anon key → Vercel env (public-safe)

### Rotation
- Rotate webhook secret every 90 days (Settings > TradingView > Rotate)
- Rotate AI keys when usage pattern changes
- Rotate broker keys on suspected compromise or every 180 days

## Deployment

### One-command deploys

- `pnpm deploy:web` → Vercel via git push
- `pnpm deploy:worker` → `fly deploy` from `apps/worker`
- `pnpm deploy:db` → `supabase db push` after migration merged

### CI/CD
GitHub Actions:
- PR → run `pnpm test`, `pnpm lint`, `pnpm typecheck`, `pnpm build`
- Merge to `main` → auto-deploy web + worker
- Nightly → Supabase schema diff alert if drift

## Latency budget

For paper trading, this is overkill but good discipline:

| Hop | Target | Max acceptable |
|-----|--------|----------------|
| TradingView → webhook | 200–800ms | 2s |
| Webhook → setup_candidate insert | < 50ms | 200ms |
| setup_candidate → risk check | < 20ms | 100ms |
| Risk check → order to Tradovate | < 100ms | 500ms |
| Tradovate fill → DB update | < 200ms | 1s |
| DB update → frontend (realtime) | < 500ms | 2s |
| **TradingView → visible on UI** | **< 2s** | **5s** |

For live with prop firms, tighten these. For NY scalping on 1-min bars, 2 seconds total is fine.

## Why not serverless-only for the worker

Serverless functions cold-start. Trading events can't wait 500ms for a cold Lambda. The worker is a persistent always-on process that:

- Maintains WebSocket to broker (reconnecting, heartbeat)
- Holds in-memory state (active setups, open positions, rate-limit trackers)
- Processes queued events (BullMQ workers)
- Runs scheduled tasks (session open/close hooks, economic calendar sync)

Fly.io machines stay warm. Cheaper than dedicated EC2, warmer than Lambda.

## What about Railway / Render / Heroku?

- **Railway**: Fine alternative to Fly.io. Slightly easier DX, similar price.
- **Render**: Works but slower cold restarts.
- **Heroku**: No. Expensive dynos, dynos sleep.

Fly.io's region selector (Chicago) is the tiebreaker for futures. If you don't care, Railway is easier.
